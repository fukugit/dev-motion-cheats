📦 Docker Image — The Blueprint Read-only

An image is an immutable, read-only template built from layered file system snapshots. Think of it as a class definition — you don't run it directly, you instantiate containers from it.

🐧 Base OS (Alpine Linux 3.19) Base Layer
⚙️ Runtime (Node.js 20) Runtime Layer
📂 Dependencies (node_modules) Deps Layer
🔧 App Config (.env, nginx.conf) Config Layer
🚀 App Code (server.js, …) App Layer
🔒 Each layer is cached. Re-builds only update changed layers — making builds fast!
# List images on your machine docker images REPOSITORY TAG IMAGE ID SIZE myapp 1.0 a3f9c12d8e01 142MB node 20-alpine b7d3e88f1c00 58.4MB # Inspect layers docker history myapp:1.0 # Push to registry (Docker Hub or ECR) docker push 123456789.dkr.ecr.ap-northeast-1.amazonaws.com/myapp:1.0
🟢 Docker Container — The Instance Running

A container is a live, isolated process spawned from an image. Multiple containers can run from the same image — each completely isolated from the others.

☁️ AWS Container Services AWS

AWS provides managed services to store, orchestrate, and run your Docker containers at scale — without managing servers.

🗄️ Amazon ECR
Elastic Container Registry
Private Docker image registry. Store, version, and scan your images. Like Docker Hub, but inside AWS.
🎛️ Amazon ECS
Elastic Container Service
Orchestrates containers using Tasks and Services. Defines where & how containers run on EC2 or Fargate.
⚡ AWS Fargate
Serverless Compute for Containers
Run containers without managing EC2 instances. Pay per vCPU & memory used. Zero server management.
