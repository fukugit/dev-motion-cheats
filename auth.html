OAuth 2.0 — Authorization Code Flow

// OAuth 2.0 Authorization Code Flow example (Next.js + NextAuth.js) // next-auth config (app/api/auth/[...nextauth]/route.ts) import NextAuth from 'next-auth'; import GitHubProvider from 'next-auth/providers/github'; export const { handlers, signIn, signOut, auth } = NextAuth({ providers: [ GitHubProvider({ clientId: process.env.GITHUB_CLIENT_ID, clientSecret: process.env.GITHUB_CLIENT_SECRET, }), ], }); // Call GitHub API with access token const session = await auth(); const res = await fetch('https://api.github.com/user', { headers: { 'Authorization': `Bearer ${session.accessToken}` }, });

JWT vs OAuth 2.0 — Usage Guide

🔑 JWT
Purpose
Securely transmit information & authenticate
Format
Signed JSON (Base64URL)
Storage
localStorage / httpOnly Cookie
  • Verifiable without server (stateless)
  • Ideal for microservice authentication
  • Can embed expiry and role information
  • Token revocation is difficult (must wait for expiry)
  • Payload tends to grow large
🔐 OAuth 2.0
Purpose
Delegate authorization to third parties
Format
Framework (token format depends on implementation)
Storage
Server-side session or Cookie
  • Enables "Social Login" via Google / GitHub etc.
  • Fine-grained permission control with scopes
  • Long-term session management with refresh tokens
  • Complex flow (multiple redirects)
  • Creates dependency on authorization server
💡 In Real Products?
In many systems, OAuth 2.0 is used to authenticate users, and JWT is used as the token format — a common combination.
Example: Login with GitHub (OAuth 2.0) → Server issues JWT → Subsequent API calls use JWT

Token/ID Storage and Transport

JS AccessNot allowed (HttpOnly protects token from XSS)
TransportBrowser auto-attaches via Cookie: header (no code needed)
CSRF ProtectionUse SameSite=Strict attribute or combine with CSRF token
Recommendation✓ Most secure. Standard pattern for production web apps
JS AccessAccessible (risk of token theft via XSS)
TransportManually set Authorization: Bearer {token} via fetch/axios
CSRFNot a Cookie, so no CSRF risk
Warning✕ Risk of token theft via XSS attack. Avoid storing sensitive data
PersistenceNone (lost on page reload or tab close)
XSS ResistanceHigher than localStorage (less directly accessible by other scripts)
TransportManually set Authorization: Bearer {token} via fetch/axios
Recommendation✓ Effective for short-term access token storage in SPAs
How it worksToken body stored in DB. Browser only holds session_id (key)
RevocationInstant logout by deleting DB record (forced revocation impossible with JWT)
ScalingHorizontal scaling requires shared cache like Redis
Recommendation✓ For systems requiring forced logout and session management
// ① Cookie (auto-send — credentials: 'include' required) fetch('/api/user', { credentials: 'include' // ← this auto-attaches the Cookie }); // ② Authorization header (for localStorage / memory storage) const token = localStorage.getItem('token'); // or let token = '...' fetch('/api/user', { headers: { 'Authorization': `Bearer ${token}` } }); // ③ Auto-attach to all requests with axios interceptor axios.interceptors.request.use(config => { const token = localStorage.getItem('token'); if (token) config.headers.Authorization = `Bearer ${token}`; return config; }); // ④ Issuing Cookie server-side (Node.js / Express) res.cookie('jwt', token, { httpOnly: true, // JS cannot read this secure: true, // HTTPS only sameSite: 'strict', // CSRF protection maxAge: 3600000 // 1 hour (ms) });
